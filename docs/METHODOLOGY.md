# Methodology

The math, assumptions, and honest limitations of the fragility analysis.

---

## The fragility model

Each lane gets a composite fragility score combining three factors:

```
fragility_score = blast_radius_pct × failure_prob × (1 / (1 + alternate_paths))
```

Higher score = more fragile. The multiplication is deliberate: a lane is only truly fragile when **all three factors align** — high impact, realistic failure scenarios, and no backup.

### Factor 1: Blast radius

For each edge `(u, v)`:

1. Count baseline reachable origin-destination pairs across the whole graph
2. Remove the edge from a copy of the graph
3. Recount reachable pairs
4. `routes_broken = baseline - still_reachable`
5. `blast_radius_pct = 100 × routes_broken / baseline`

Implementation uses `nx.has_path()` which runs BFS internally. Total complexity is O(V × E) per simulation × E edges = O(V × E²). At branch scale (≤ 200 lanes), this runs in seconds.

### Factor 2: Failure probability (heuristic)

```python
def estimate_failure_probability(mode, transit_days):
    base = 0.08 if mode == 'Air' else 0.05
    if transit_days and transit_days > 10:
        base += (transit_days - 10) * 0.005
    return min(base, 0.30)
```

**Logic:** Air lanes have a slightly higher baseline failure rate (capacity cancellations, slot bumps, weather diversions). Long sea routes accumulate exposure to weather and port congestion proportional to transit days. Cap at 30% — even the most disrupted lanes don't fail more than ~1 in 3 in real operations.

**Limitation:** these numbers are educated guesses, not empirical estimates. In production, replace with:
- Carrier historical on-time % from TMS/booking system
- Weather incident counts on the route
- Port congestion indices (e.g., from S&P Global Platts, Kuehne+Nagel SeaExplorer)
- Capacity utilization signals (carrier load factor)

The heuristic gives **relative ordering** that's directionally correct. It should not be cited as an absolute prediction.

### Factor 3: Redundancy

```python
def count_alternate_paths(graph, u, v, cutoff=4):
    G_copy = graph.copy()
    if G_copy.has_edge(u, v):
        G_copy.remove_edge(u, v)
    return len(list(nx.all_simple_paths(G_copy, u, v, cutoff=4)))
```

After removing the direct edge, count all paths from `u` to `v` with length ≤ 4. The cutoff prevents exponential blowup on dense graphs and reflects operational reality — nobody routes cargo through 5+ waypoints.

**Important limitation:** in this dataset, every edge has `alternate_paths = 0` because the graph contains only the branch's own lanes. A realistic model would add:
- Partner-carrier routes (other forwarders this branch has agreements with)
- Competitor-market routes (the customer's option to defect to another forwarder)
- Trans-shipment options through nearby ports

When `alternate_paths = 0` uniformly, the redundancy term collapses to a constant `1/(1+0) = 1` and the score is effectively `blast × probability`. The ranking is still valid but the scale is dampened. Add real alternate-path data and the score becomes more discriminating.

## Network metrics — quick reference

| Metric | What it measures | Range |
|---|---|---|
| `nx.density(G)` | Fraction of possible edges that exist | 0 (sparse) to 1 (fully connected) |
| `nx.degree_centrality(n)` | Direct connections to/from this node | 0 to 1 (or higher in directed graphs) |
| `nx.betweenness_centrality(n)` | Fraction of shortest paths passing through this node | 0 to ~1 |
| `nx.closeness_centrality(n)` | Average inverse distance to all reachable nodes | 0 to 1 |
| `nx.eigenvector_centrality(n)` | Connection to other well-connected nodes | 0 to ~1 |

Different definitions answer different questions. A node can rank #1 on degree and last on betweenness — that's not a bug, that's the point.

## Where this would scale to next

If this work were brought into production at a real branch:

1. **Auto-ingest from TMS** — replace the CSV with a daily extract from CargoWise, Descartes, or whichever TMS the branch operates
2. **Real carrier data** for the failure probability model
3. **Multi-branch view** — extend the graph to include the global network, exposing inter-branch transshipment options
4. **Continuous monitoring** — re-run nightly, alert on lanes whose fragility score crosses a threshold
5. **Interactive Streamlit/Dash front-end** — expose the model to operations leadership without requiring a notebook user

Each step is straightforward; the methodology developed here ports unchanged.
